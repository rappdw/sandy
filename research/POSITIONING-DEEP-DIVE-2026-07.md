## 1. Executive summary

**Positioning verdict (the one paragraph):** Sandy is the only tool that gives an individual developer a *per-project, credential-isolated, whole-session* sandbox that wraps **any** coding agent (Claude/Gemini/Codex/OpenCode, alone or side-by-side) in one hardened container with one uniform egress proxy — and defends against a *malicious committed `.sandy/config`*. That combination is still unique in July 2026, and the axes it rests on (per-project agent-state + credential sandboxes; cross-agent uniformity; multi-agent-in-one-box) are **structurally off the roadmap of every vendor** and every "platform" competitor. **The niche is not closing where sandy is strong.**

**But the ground moved in three ways since the June synthesis:**

1. **The kernel-boundary tier is being commoditized.** Docker Sandboxes (`sbx`) GA'd ~Mar–Apr 2026: real per-agent microVMs, cross-platform *including Windows*, free for commercial use, same agent set as sandy — and it now advertises host-side credential injection, per-sandbox egress policy, and file scoping. It out-classes sandy on the one boundary sandy can't match (shared kernel) and contests three of sandy's security features simultaneously. Sandy cannot win the "strongest boundary" race and should stop implying it can. (Confidence: H on what Docker Sandboxes is; M on exact feature parity.)
2. **Two features became table stakes and sandy trails on both.** *L7 method/path egress filtering* and *proxy-side credential brokering* each now appear in **4+ independent new launchers** (mattolson/agent-sandbox, clawker, nono, agentbox; OpenShell, Devin CLI). Sandy has neither. The broker is already sandy's own #1 POST_1.0 item; L7 is correctly rejected on the TLS-termination constraint but the *pressure* is now real and multi-sourced.
3. **Vendors are moving onto the egress + credential axes — but incompatibly.** Codex now defaults to bwrap+seccomp and *documents running `--sandbox danger-full-access` in Docker* (sandy's exact choice, now vendor-blessed); Gemini reversed to egress-off-by-default; Claude Code shipped `sandbox.credentials` (bash-subprocess credential shielding) *after* our pin. Each is agent-specific and differently-defaulted, so a developer running three agents gets three isolation stories — the seam sandy fills. This *strengthens* the uniform-wrapper story today, with a named erosion trigger (container-native mode + credential brokering maturing across 2–3 major agents).

**Most actionable own-finding (sandy's own bug):** `SANDY_EGRESS_PROXY` sits in `SANDY_PASSIVE_KEYS` under the comment "No security-boundary impact" (`sandy:51`), but a committed workspace `.sandy/config` can set it to `0` — disabling network isolation entirely on macOS (no iptables fallback) — with **no approval prompt**, overriding even a user's defensive host-config `=1` (only a shell `export` outranks a passive workspace value). The array violates its own stated invariant. Fixable as an invariant-bugfix (asymmetric tiering: allow *tightening* → `2` from a workspace, gate *loosening* → `0` behind approval), which sidesteps the 1.x "no retiering" rule. (Confidence: H — verified against the key array and the CLAUDE.md macOS-limitation section.)

**Top adopt recommendations (detail in §6):** (1) close the `SANDY_EGRESS_PROXY` passive-downgrade gap [1.0.x]; (2) `.env`/secret-file protection — the long-standing "highest-value remaining" gap, now reinforced by srt, Gemini history, and Codex all touching it [1.0.x/1.1]; (3) a **transparent-launch primitive** (`SANDY_LAUNCH=raw`/`sandy --exec`, no inner tmux) — the single change that turns the huge worktree-manager ecosystem (vibe-kanban 27k★, cmux 23.5k★, Superset, Orca…) from bystanders into point-and-harden front-ends for sandy [1.1]; (4) document CC-nested-in-sandy (`enableWeakerNestedSandbox` + `sandbox.credentials`) as the near-free way to shield mounted creds + workspace `.env` from prompt-injected bash [1.0.x doc]; (5) the host-side git credential broker, agentbox's git-shim design over a **bind-mounted Unix socket** (not their `host.docker.internal` HTTP, which sandy's proxy blocks) [1.1]; (6) pin the gstack skill pack to a resolved SHA — sandy currently auto-rebuilds onto garrytan/gstack's unpinned `main` HEAD [1.0.x].

**Anti-roadmap (reaffirmed):** no L7/TLS-MITM, no deny-first default, no platform/control-plane, no persistent-service mode, no driver/backend abstraction, no per-command escape hatches. Every "platform" peer traded away the hardened-container edge; sandy's single-file minimalism is the differentiator, not a stage to grow out of.

## 2. The field, mapped

### 2.1 Taxonomy (the six sweep categories collapse to four strategic groups)

- **Group A — Per-project agent sandboxes (sandy's genre).** Wrap an existing agent CLI in an isolated environment scoped to one project. sandy, bulkhead, mattolson/agent-sandbox, clawker, streamingfast/sbox, RchGrav/claudebox, the claudebox family. *Competitive core.*
- **Group B — Vendor-native sandboxing.** The agent sandboxes *itself* (bash-command scope). Claude Code `sandbox.*`, Codex bwrap/Seatbelt, Gemini sandbox modes, Copilot MXC, Devin `--sandbox`. *Substitution threat for the single-agent case.*
- **Group C — Session/fleet managers & platforms.** Orchestrate many agent sessions, add isolation as a feature. agentbox, OpenShell, Docker Sandboxes, alice, container-use, conductor, and the manager set. *Adjacency: some compose with sandy (channel partners), some bundle rival isolation (competitors).*
- **Group D — Isolation engines / boundaries.** Primitives, not products. sandbox-runtime (srt), gVisor, Kata, hyperlight, libkrun, microVM VMMs. *Ingredients sandy could ride (`--runtime`), not adopt wholesale.*

### 3.1 alice (jcronq/alice) — *diverging sibling; low convergence risk*

Persistent personal-agent runtime: one long-lived container (`restart: unless-stopped`, `stop_grace_period: 30m`), s6-supervised Signal daemon + speaking/thinking hemispheres, git-backed Obsidian memory vault, label-driven idea→PR state machine. **No egress control; publishes 3 host ports** (7777/8080/8765). It is nearly the *inverse* of sandy on lifecycle and network posture. (Local pin 2026-06-07; upstream tip 2026-07-01, ~29 commits, **100% mind-side — zero isolation/security commits**; the two projects are diverging, confirmed across two commit listings.)

- **Division of labor is stable (H):** alice = autonomous-agent *lifecycle*; sandy = per-session *isolation substrate*. "alice-in-sandy" is a poor fit today (sandy exits when the agent exits; no `docker exec` surface; zero published ports; egress lock blocks Signal + host-gateway LiteLLM). Better framing: sandy could isolate the detached `claude` subagents alice's dispatcher spawns, not host alice itself.
- **The June HANDOFF_TO_SANDY had 0-of-4 uptake (H, independently verified against the sandy script):** torn-read fix on `.claude.json` — *not shipped* (`sandy:~5018` still single `JSON.parse` → `exit(1)` → `|| cp` copies the same garbage bytes; unchanged since 2026-03-27); credential dir-mount — deferred (but its trigger has since fired, see below); `sandy send` — not shipped; `events.jsonl` — not shipped. Post-June sandy effort went entirely to the rc cut + proxy resilience + Claude OAuth + M4 stabilization.
- **New inbound candidate (M-H):** alice's `SseEventLogger` (`src/alice_cozylobe/event_log.py`) is a *hardened* event-log reference (lazy-open, date-rotation, warn-once, injectable clock, fail-safe swallow) — the shape to copy *if* sandy ever builds `events.jsonl`, over the naive jq-append.
- **A sharp finding for the reverse handoff:** alice ships weak default creds (`sk-alice-local`, `admin:alice-admin`) on a LAN-published LiteLLM dashboard (`docker-compose.yml:204-229`) — exactly what sandy's config-tier/secret discipline exists to catch.

**Take:** the torn-read fix (it's sandy's own June ask, still open) + the `SseEventLogger` design if an event log is ever built. **Ignore:** persistent-daemon lifecycle, published ports, label-driven SM, Signal/Gmail transports — the autonomous-agent niche sandy deliberately isn't in; a "service mode to host alice" would break the ephemeral-sibling identity. **Already better:** egress isolation (alice has none), protected-file enforcement (alice's is documentation), credential ephemerality (alice symlinks host creds into a long-lived container), config-tier secret discipline, workspace mutex + per-project sandboxes.

### 3.2 bulkhead (pmembrey/bulkhead) — *dormant project, three donated ideas + one gift-in-reverse*

Rust CLI generating a `devcontainer.json` and delegating lifecycle to the Dev Container CLI. **Effectively dormant** (local pin = upstream tip, both 2026-04-27; 2★, self-described unstable). Study the ideas, not the competitor.

- **Namespace-flag guard (mechanism, H):** `validate_run_args()` hard-`bail!`s on `--privileged`, non-allowlisted `--cap-add`, `--mount/-v`, `--device`, `--security-opt`, and host-namespace joins — *no override, no consent path*. Stronger than a re-promptable approval for those specific flags (matches sandy's own "hard-error > trainable y/N" philosophy from symlink approval), but guards only one config surface. Sandy's tiering guards a far broader set.
- **This surfaced sandy's own gap (H — the headline):** see §1 and §6. `SANDY_PASSIVE_KEYS` claims "no security-boundary impact" but includes `SANDY_EGRESS_PROXY` (loosenable to `0` = isolation off on macOS) and `SANDY_ALLOW_WORKFLOW_EDIT` (drops `.github/workflows` from `:ro`). Fixable as an invariant-bugfix.
- **Supply-chain env defaults (M):** `PYTHONDONTWRITEBYTECODE=1`, `PIP_DISABLE_PIP_VERSION_CHECK=1`, `UV_LINK_MODE=copy` — zero-breakage hygiene sandy sets none of. `NPM_CONFIG_MINIMUM_RELEASE_AGE=1440` (no npm package < 24h old) is a novel zero-day-npm defense but must be scoped to agent-runtime, not the image build (would stall sandy's own same-day CC updater). `NPM_CONFIG_IGNORE_SCRIPTS=true` is too breakage-prone as a blanket default — bulkhead itself disables it for its own bootstrap.
- **Clone mode (M):** the agent works in a `git clone --no-local` under `.bulkhead/clones/`, so the source `.git` is never in the container mount namespace — injection into the *source* repo is structurally impossible. But it's opt-in, full-copy, with manual sync-back; sandy's default `:ro` git-internals mounts + session-end detection cover the same vectors transparently.
- **devcontainer-CLI delegation validates, doesn't threaten:** it adds a hard Node.js host dep and cedes `docker run` flag control, so it *cannot express* sandy's egress-proxy sidecar, attestation marker, ephemeral cred mounts, or workspace mutex. Hand-rolled `docker run` is correct for sandy's pillars.

**Take:** the three zero-breakage env defaults [1.0.x]; `MINIMUM_RELEASE_AGE` as an opt-in agent-runtime knob [1.1]; **fix sandy's own passive-key gap** [1.0.x]. **Ignore:** clone mode, devcontainer delegation, blanket `IGNORE_SCRIPTS`. **Already better:** egress-by-default, credential ephemerality (bulkhead forwards keys into a *persistent* `~/.claude` volume), multi-agent, broader untrusted-config defense.

### 3.3 agentbox (madarco/agentbox) — *the credential-broker reference design*

Very active (local v0.15-era 2026-06-06; upstream v0.21.0 2026-06-30). The broker architecture is unchanged upstream, only accreting. **This is the single most valuable design to borrow.**

- **Broker mechanism (precise, H):** four parts. (1) PATH-shadowing `git`/`gh` shims intercept *only* the network ops (push/pull/fetch/clone), with a **strict per-op flag whitelist** (push allows only `--force-with-lease`, etc.; unlisted flag → die) and **positional refusal** (remote/branch rebuilt host-side, never from box argv); non-network git falls straight through. (2) Transport over HTTP to a detached host relay via `host.docker.internal:8787`. (3) The op is **executed host-side** with the user's real creds (`git -C <hostWorktree> …`, `process.env` inherited for `SSH_AUTH_SOCK`/gh helper) — **only stdout/stderr/exitcode cross back; the credential never enters the container.** (4) Per-*action* host-side approval showing exact argv+cwd; read-only ops bypass; loopback-guarded admin endpoints (a box cannot answer its own prompt). Sharpest scar-tissue detail: the one-time host-initiated token is passed as a **CLI arg, not an env var**, specifically because env vars are inherited by child processes (a pre-push hook could replay one).
- **Critical adaptation for sandy (H, verified `sandy:5389`):** agentbox's `host.docker.internal` HTTP transport **is blocked by sandy's `--internal` sidecar by default** — porting it verbatim fights sandy's own posture. The POST_1.0 sketch's **bind-mounted Unix socket is strictly better** and should stay: fail-closed by construction, no bearer token, no loopback-guard needed. Sandy is also *simpler* than agentbox here — it single-bind-mounts the workspace (host `.git` and container worktree are the same inode), so there's no worktree-map to resolve: one workspace = one `git -C $WORK_DIR` helper.
- **Checkpoints/snapshots (H): ignore.** `docker pause`/`unpause` + `docker commit` warm-start, built to manage *many concurrent boxes per project* and auto-pause idle ones. Sandy's workspace mutex enforces one session per workspace — nothing to switch between — and its persistent package caches already give warm deps without a commit step.
- **Multi-agent dashboard (M):** a PTY compositor for *N boxes across projects* — a different axis from sandy's *intra-workspace* panes. The one genuinely-absent companion worth noting: an **unattended-orchestrator approval surface** (`agent approvals`/`approve <id>` over a machine API), which is the natural companion to the broker — a JSON pending-approval list on `--print-state`.

**Take (high):** the broker pattern — shim (flag-whitelist + positional-refusal) + bind-mounted-socket host helper + per-action content-bound approval [1.1]; and a machine-readable pending-approval list on `--print-state` *if* the broker ships [1.1]. **Ignore:** checkpoints/pause-resume, cloud teleport (reaffirms June "ignore multi-cloud"), PTY dashboard. **Already better:** fail-closed Unix-socket transport vs their HTTP-over-gateway; egress-by-default (agentbox has none on the box); simpler single-mount broker surface; credential ephemerality already present.

### 3.4 The existential trio: Claude Code + sandbox-runtime (srt) + CC docs

The decision-relevant section. (claude-code pin 2026-06-06 = CC v2.1.168; live tip 2026-06-XX = CC v2.1.199, fetched 2026-07-03.)

- **CC shipped credential shielding *after* our pin (H — the headline delta):** the pin has only `CLAUDE_CODE_SUBPROCESS_ENV_SCRUB`; between pin and today CC added **`sandbox.credentials`** (v2.1.187: deny credential *files* + unset secret *env* for sandboxed bash), per-session remembered network allows (v2.1.191), and experimental `credentials.mask` + `network.tlsTerminate` (v2.1.199). This *is* the move onto sandy's credential turf.
- **But verify what it is (H):** per the official docs, `sandbox.credentials` is a **deny/mask list for sandboxed *Bash commands only*** — "affects sandboxed Bash commands only." It protects creds *from the subprocess*; it is **not** per-project credential *partitioning*. CC still uses one global `~/.claude/.credentials.json`.
- **Per-project isolation — is April's "CC has none" still true? Refined-yes (H):** sandbox *policy* is now per-project-configurable (`/sandbox` writes `.claude/settings.local.json`). But per-project agent-*state*, credential-*sandbox*, and package-*caches* do not exist and show **zero trajectory** — a vendor has no product reason to isolate the agent from *itself* across projects (median CC user runs one repo).
- **Durable-moat stress test (H unless noted):** *whole-session isolation* — CC's sandbox is **Bash-tool-ONLY** ("Read, Edit, Write, WebFetch, MCP use the permission system"); srt *can* wrap the whole process but that's a DIY primitive with no state/cred/multi-pane. **Structurally durable.** *Cross-agent uniformity* + *multi-agent panes* — no vendor will ship Gemini/Codex panes. **Durable.** *Untrusted-repo protected files* — srt's `DANGEROUS_FILES`/`DIRECTORIES` list is a near-mirror of sandy's but **bash-only** and misses `.github/workflows`, `.circleci`, `.devcontainer`, `.pre-commit-config.yaml`, submodule gitdirs; conversely srt *prevents* absent-path writes (via `/dev/null` mount) where sandy is detection-only. **Converging (M).** *Attestation* — CC is the attestee, has none. **Durable, niche.**
- **Composition — CC sandbox nested inside sandy (M, couldn't run Docker to confirm):** CC's Linux bwrap sandbox fails to mount a fresh `/proc` in an unprivileged container; the docs' `enableWeakerNestedSandbox:true` bind-mounts the container's existing `/proc` and describes sandy's setup *exactly* ("Only use this when the outer container already provides the isolation boundary"). Sandy's flags (`no-new-privileges`, `cap-drop ALL`, default seccomp, no userns remap) should permit unprivileged userns creation. **Verdict: tolerate + document opt-in; do not enable-by-default, do not disable.** Nested *isolation* is marginal (sandy is already the boundary), but the nested fs-write allowlist + `sandbox.credentials` add **real defense-in-depth the claude pane lacks today** — shielding sandy's own mounted creds and workspace `.env` from prompt-injected *bash subprocesses* (open thread #1).
- **Mechanism vs srt:** srt is TCP-only, SNI/hostname allowlist, default no-TLS-terminate (like sandy) with *optional* MITM/L7 as a toolkit that "does not bless any matching DSL"; Linux removes the netns entirely and routes via unix sockets (non-TCP fails closed, same guarantee as sandy's `--internal`, different mechanism). srt's escape-hatch culture is rich (`dangerouslyDisableSandbox`, `allowUnsandboxedCommands`, `enableWeaker*`) — philosophically opposite to sandy's no-per-command-hatch stance.

**Take:** `sandbox.credentials`-style bash-subprocess cred shielding — cheapest path is *documenting* CC-nested-in-sandy [1.0.x doc], native env-scrub for non-claude panes [1.1]; the `failIfUnavailable` framing validates `SANDY_FAIL_IF_NO_ISOLATION=1` [1.0.x]; note srt's absent-path *prevention* against sandy's fanotify roadmap. **Ignore:** L7/TLS-MITM (srt ships no DSL, calls SNI-only acceptable), per-command escape hatches, Windows. **Already better:** whole-session isolation, cross-agent uniformity + multi-agent, per-project state/cred/cache sandboxes (CC has none, no trajectory), protected-file breadth, attestation. **One-line strategic read:** CC/srt are perfecting the *bash-command* sandbox and just landed subprocess credential shielding sandy lacks — but the axes sandy owns are structurally off vendor CC's roadmap. **The niche is not closing; the one urgent borrow is bash-subprocess credential shielding, and CC-nested-in-sandy is the near-free way to get it.**

### 3.5 OpenShell (NVIDIA) — *verdict splits: ignore what it builds, watch what it is*

The most active repo in the set (936 commits, tip 2026-07-03, near-daily releases, **7.4k★ +~1.7k in 2mo**, GTC-2026 launch with a Jensen Huang quote), but still all `0.0.x`/alpha with no named production adopters. Institutionally sold as an enterprise agent-security platform; shipped software is honest solo-dev alpha.

- **Individual-dev niche (H): still platform-shaped.** `curl|sh` installs a *daemon* — every workflow needs a reachable, persistent gateway with SQLite/Postgres, mTLS PKI, JWT, gRPC control plane. Opening egress = authoring YAML (the minimal one-endpoint GitHub policy re-specifies the entire filesystem/landlock/process block). vs sandy's `cd project && sandy`, zero YAML, permissive-just-works. Their own "single-player mode" is a *reduction of a platform*, not a native laptop tool.
- **L7 + hot-reload (H): mechanism is TLS-MITM.** An ephemeral CA injected into the container trust store terminates TLS per-hostname; OPA/rego policy + a Z3 SMT prover model the exfil distinction (POST/PUT flagged as capability-expansion). **Cost to sandy: not justified.** It breaks sandy's never-terminate-TLS invariant, and for OpenShell TLS-termination is load-bearing for *three* features at once (L7 policy + credential injection into HTTPS bodies + inference rerouting) — sandy brokers no creds and reroutes no inference, so MITM buys it *only* method/path filtering; the plaintext-only middle path is near-worthless since ~all agent traffic is HTTPS. The prior SKIP verdict *strengthens* with better evidence.
- **Deny-first vs permissive (H):** the strongest evidence *against* deny-first for a solo tool is how much machinery OpenShell built to survive it — a full policy-advisor subsystem (L4 mapper + agent-authored proposals + Z3 prover + approval gate + structured 403s) so the agent can self-author rules while it "sleeps on a socket… usually under two minutes" per new endpoint. Elegant, but a human-in-the-loop interrupt per endpoint; a coding agent touches npm/pypi/crates/github/model-API/docs constantly. Sandy's permissive default (zero friction, closes the LAN/metadata gap) + strict-as-opt-in is correct for its user; the exit-time `SANDY_ALLOW_HOSTS` hint is the right-sized analog of the deny→proposal loop.
- **NVIDIA weight (H signals, M impact):** real and flagship-level, but the 7.4k stars are largely NeMo-halo, not `sandbox create` adoption. Changes *enterprise* dynamics (never sandy's niche), not yet solo-dev. **Watch-trigger:** OpenShell ships a daemonless, zero-YAML, permissive-capable laptop mode.
- **Backend pluggability → `SANDY_RUNTIME` (one line):** OpenShell's `ComputeDriver` trait (incl. a libkrun microVM driver) validates the *pattern* and prices the *cost* ("first run takes a few minutes," 10-30s boot, its own crate). Lesson: take the opt-in-stronger-boundary idea, **reject the driver abstraction** — ride Docker's existing `--runtime` flag, don't build a backend layer.

**Take:** structured deny→proposal 403 with agent guidance (already partially adopted); checksum-verified upgrade + breaking-upgrade ack (version-hygiene); the pluggability *pattern* → `SANDY_RUNTIME` as a `--runtime` passthrough; credential-broker *inspiration* (but theirs needs MITM, so sandy's must be host-side git-shim). **Ignore:** gateway/gRPC/mTLS control plane, L7+MITM, Z3+OPA+YAML policy, deny-first default. **Already better:** zero-config UX, single-file auditability (6.2k bash + 2.2k Go vs ~185k LOC/17 crates), no persistent server-side state, egress-on with zero friction. **Verdict: June "ignore" stands on what sandy *builds*; tightens to an active *watch* on the competitive axis; does not reverse.**

### 3.6 Boundary options: hyperlight + gVisor + Docker Sandboxes

- **hyperlight (ignore, confirms prior, H):** a Rust micro-VM *library* for embedded function-as-a-service isolation (µs guest-function calls, **no guest kernel/OS/syscalls/fs/net**). Explicitly not for full Linux workloads — the exact surface sandy needs. Nothing transfers beyond philosophy.
- **`SANDY_RUNTIME=runsc` (gVisor) — cheap code, expensive to *own* (H on cost, M on the load-bearing risk):** the launcher change is genuinely small (validate + detect registered runtime + one `--runtime` flag, fail-closed if absent). But the *feature* is a recurring, Linux-only compatibility-soak liability: gVisor's netstack has known Docker-embedded-DNS breakage on user-defined bridges (whether sandy's proxy `--dns`/SNI/CONNECT path survives netstack is exactly what only a soak proves — **the load-bearing risk**); Docker v29 currently fails overlayfs under gVisor (open upstream), and sandy auto-rebuilds on Docker bumps so it'd inherit such breakage silently; ~10-30% perf tax lands on sandy's build/test/Playwright workloads; **macOS is out entirely** (Linux-only). It closes R1 (shared-kernel escape) — a **low-probability threat for sandy's actual adversary** (a wrong-but-not-evil prompt-injected agent has no kernel 0-day). **Ship opt-in, Linux-first, privileged-tier, experimental, gated on the network soak; never a default.**
- **Docker Sandboxes — the real strong-boundary competitor (H on what it is; M on exact gap; feeds existential (b)):** GA'd ~Mar–Apr 2026 (the April doc's flag was correct — it shipped). **Real per-sandbox microVM** (own kernel, FS, network, private inner Docker daemon), custom cross-platform VMM (macOS Hypervisor.framework / Windows WHP / Linux KVM), local-first, CLI `sbx run claude`, **free including commercial** (only org governance is paid), same agent set as sandy. Docs claim host-side credential injection, per-sandbox network policy, and file scoping — **it contests sandy's ephemeral-creds + egress-allowlist + protected-file edges *while adding the VM boundary sandy lacks*, cross-platform incl. Windows, free.** What sandy plausibly still holds (M): the malicious-committed-`.sandy/config` threat model (no evidence Docker addresses it), multi-agent side-by-side in one box (Docker appears one-µVM-per-agent), single-file/no-daemon auditability, attestation, and egress *granularity* (SNI/CONNECT/strict-allowlist/LAN-block vs less-documented "network endpoints").
- **Kata / microsandbox / Arrakis / e2b / Daytona / Modal (watch/dismiss):** Kata = heavier VM-per-container, possible `SANDY_RUNTIME=kata-runtime` alongside runsc, same Linux-only caveat. microsandbox/Arrakis = local-first microVM but for *agent-generated code via SDK/REST*, a different product (local e2b analog) — watch microsandbox. e2b/Daytona/Modal = cloud SaaS for agent-generated code — different niche, dismiss (reaffirms June).

**Strategic read:** the stronger-kernel-boundary tier is being **commoditized by Docker Sandboxes** — cross-platform, free, same agents — which a shared-kernel container + runsc bolt-on cannot match. gVisor is worth shipping *only* as a defense-in-depth checkbox for the Linux security-maximalist who won't leave sandy. **Sandy's durable position must rest on its other edges, not on owning the kernel-boundary tier** — and it should say so honestly in docs rather than imply a boundary it can't hold.

### 3.7 gstack (rappdw fork of garrytan/gstack) — *the integrated skill pack; a supply-chain finding*

Not a sandbox — the skill pack sandy hosts. Local pin frozen at 2026-03-27 (fork VERSION 0.12.11.0); **upstream garrytan/gstack ran to VERSION 1.58.5.0 by 2026-06-25** (~90 days, 0.12→1.58 ahead) and publishes **zero release tags**.

- **The "pin to release tag" thread is impossible as written (H):** upstream has no tags/releases; sandy already reflects this (`SKILL_PACK_TAG_PREFIXES=("")` → falls to latest-commit-SHA). The fix must be a **sandy-side SHA pin**, not a request for upstream tags.
- **Supply-chain exposure (H, verified `sandy:132,185`):** `SKILL_PACK_REPOS` points at garrytan *upstream* (not the fork), and `skill_pack_resolve_versions()` curls `commits?per_page=1` every launch, taking `main`'s HEAD SHA; on change it regenerates `Dockerfile.skills` and auto-rebuilds Phase 2.5b (`bun install && bun run build`) onto that SHA, then symlinks its skills + puts 18 `bin/gstack-*` scripts on PATH. **Sandy tracks an unpinned third-party branch HEAD** — a bad/malicious upstream commit is silently pulled, built (build-time RCE surface), and run (runtime surface) on the next gstack-enabled launch. The 5s timeout + cache protect availability, not integrity; container isolation is the only backstop.
- **Non-Claude panes (M):** upstream *already* ships a codex transpile (`setup --host codex` → `~/.codex/skills/gstack`), so codex support is "wire up `setup --host codex` + mount," **moderate**; a faithful Gemini TOML port is manual, **hard** (gemini isn't a `--host` target). sandy's `/ss` skill precedent is a thin wrapper and not representative of gstack's 29 deeply-coupled SKILL.md files.
- **Judgment (M):** heavy, security-relevant machinery (154 skill mentions, 4 functions, 2 Docker phases, version resolution, state mount/migration) amortized over **exactly one opt-in pack** whose own reference fork froze at 0.12 while upstream ran to 1.58 → **maintenance liability > differentiator today** at N=1.

**Take:** pin the pack to a resolved SHA (or the fork) with an explicit update gate — closes the auto-rebuild-onto-upstream-HEAD supply-chain hole [1.0.x]. **Ignore:** the impossible "pin to release tag" as literally phrased. **Already better:** n/a (this is hosted content, not a competitor).

## 5. Positioning analysis (Part 3)

### 5.1 Unique position

Restated for a stranger to repeat: **sandy is the per-project, credential-isolated, whole-session sandbox that wraps any coding agent — or several side-by-side — in one hardened container with one uniform egress proxy, and is the only one that defends against a malicious committed repo config.** No other tool occupies all of that at once.

### 5.2 Top differentiators (stress-tested, not rubber-stamped)

1. **Per-project agent-state + credential sandboxes ("a venv for your agent").** *Evidence:* `~/.sandy/sandboxes/<name>-<hash>` with per-project caches + ephemeral RO credential mounts (CLAUDE.md, SPEC App. E). *Nearest substitute:* every vendor's per-project *config* (trust_level, project JSON) — but **no vendor and no Group-A competitor ships isolated per-project state+credentials**; Gemini has literally none, CC has one global `~/.claude`. *Durability:* **High** — structurally off vendor roadmaps (the median agent user runs one repo, so vendors have no reason to build it).
2. **Multi-agent side-by-side in one hardened box.** *Evidence:* 2–4 tmux panes, `sandy-full` image. *Nearest substitute:* agentbox's cross-*box* dashboard (different axis) and Docker Sandboxes' one-µVM-per-agent (not side-by-side in one box). *Durability:* **High** — no vendor will ship rival agents' panes; managers that add it become channel partners, not replacements.
3. **Config-tier defense against a malicious committed `.sandy/config`.** *Evidence:* parse-not-source + privileged-key per-workspace approval (CLAUDE.md Config tiers). *Nearest substitute:* bulkhead's namespace-flag guard (narrower, one surface). *Durability:* **High** — but see the `SANDY_EGRESS_PROXY` hole in §6; the *mechanism* is a genuine edge only if its own invariant holds.

Secondary, still real: on-by-default cross-platform egress proxy that closed the macOS gap with zero friction (uniform where vendors are fragmented); single-file auditability + spec/test/rc discipline; the self-attestation marker (the only in-container "am I really sandboxed, and at what level" trust root in the field).

### 5.3 Where sandy is structurally behind

- **Kernel-grade boundary — behind, and now commoditized.** Docker Sandboxes gives cross-platform microVMs free. **Response:** offer `SANDY_RUNTIME=runsc` as an opt-in Linux checkbox, but *do not* compete on this axis; **say so honestly in docs** ("process + filesystem + network isolation on a shared kernel; for a hardware boundary use a VM runtime or Docker Sandboxes"). This is a "behind and should say so" — not a "must respond."
- **Workspace-secret protection (`.env`) — behind, and must respond.** srt, Gemini (historically), Codex, and OpenCode all touch this; sandy leaves R2b open. Cheap to close. **Must respond** [see §6].
- **L7 egress granularity — behind, and fine.** 5+ projects have it; sandy deliberately doesn't (TLS-termination constraint). The July sweep sharpened this into a *defensible* position rather than a gap: **no project enforces method/path policy without terminating TLS** (Coder Boundary, OpenShell, clawker, mitmproxy all inject a CA), so sandy's SNI-demux is the **strongest posture available under the never-terminate-TLS invariant** — L7 is not "missing," it is *the thing you give up TLS-integrity to get*. For sandy's "occasionally-wrong, not adversarial" agent the ROI is poor. **Behind and fine (different threat model)** — document the reasoning (cite Coder Boundary as the reference alternative) so it reads as a deliberate constraint, not an oversight.
- **GUI/fleet UX — behind, and fine.** Sandy is a CLI; managers own the GUI. Better as composition (managers launch *through* sandy) than to build. **Behind and fine.**
- **Windows — absent, and fine for the niche** (Docker Sandboxes covers Windows users who need it).

### 5.4 Existential scenarios (likelihood × impact)

- **(a) CC-native gains per-project isolation + credential scoping.** *Likelihood: Low; Impact: High.* Evidence says CC is perfecting the *bash-command* sandbox and added subprocess credential *shielding* — but per-project *partitioning* of agent state/creds has **zero trajectory** and no product rationale for a single-repo-median vendor. **Response:** keep leaning into the multi-agent + per-project axes; **watch signal:** any CC setting that scopes `~/.claude` state or credentials per-project.
- **(b) Docker Sandboxes (or an OS vendor) makes hardened agent containers a checkbox.** *Likelihood: High (already shipping); Impact: High.* This is the live threat. **Response:** don't fight on the boundary; win on per-project cred/state isolation, multi-agent, committed-config defense, single-file auditability, and egress *granularity/uniformity* — and consider whether sandy could *run on top of* `sbx` as a backend someday (compose, don't compete). **Watch signal:** `sbx` adding multi-agent-in-one-box or committed-config threat modeling.
- **(c) Every vendor ships good-enough native sandboxing, fragmenting the "uniform layer" value.** *Likelihood: Medium; Impact: Medium.* Today fragmentation *helps* sandy (three agents = three incompatible isolation stories). It flips to *hurting* sandy only if a container-native mode + credential brokering matures across 2–3 major agents simultaneously. **Response:** the uniform egress proxy + one credential model + one per-project sandbox is the pitch; ship the broker to stay ahead. **Watch signal:** two+ vendors shipping container-native + broker in the same quarter.
- **(d) The multi-agent pattern fades or explodes / a manager becomes "sandy plus a GUI".** *Likelihood of a sandy-grade manager: Low at current velocity; Impact: High.* The sweep is decisive: the high-star managers (vibe-kanban 27k, cmux 23.5k, Superset 12k, Orca 11.5k, conductor) are **all worktree/none with zero isolation-hardening roadmap evidence**, and the isolation leaders (agenttier, container-use) lack desktop UX and are tiny/stalled. Nearest by *ambition* is Sculptor (funded, markets "containers that keep your machine safe") but it **ships worktree-by-default and is trending away from containers**. The realistic mechanism to watch is a high-star worktree manager bolting on a *plain* container backend — parallel-code already demonstrates the pattern (optional `Dockerfile`) and cmux has an open request — but *plain* containers ≠ sandy's egress-proxy + ephemeral-creds + read-only wedge, which is on **no surveyed manager's roadmap**. ETA for plain containers in a 10k★ manager: ~6–12 months plausible; sandy-grade convergence: **>12 months, low probability**. **Response — the strategic play:** make sandy the *isolation layer managers compose with* by shipping the transparent-launch primitive (§6, rec #3) so the whole worktree-manager field becomes point-and-harden front-ends for sandy, not rivals. **Watch signals:** Sculptor re-adding per-agent containers *with a network policy*; the parallel-code optional-Dockerfile pattern spreading to a high-star manager.

### 5.5 Target user — revisited

The working hypothesis (an individual dev running one+ agents with real creds on real projects, macOS/Linux, wants isolation without ceremony) **holds and is well-served** (H). Two adjacencies sandy serves *almost for free*: (1) **the security-conscious solo dev / small team** running agents against *untrusted* repos — the committed-config + protected-files + attestation story is already theirs; worth naming explicitly in the README. (2) **CI/headless** — sandy's `-p` + `SANDY_AUTO_APPROVE_PRIVILEGED` already work; a light "CI recipe" doc would capture it. Neither justifies platform-ification; both are documentation, not features.

### 5.6 The anti-roadmap (reaffirmed / revised)

Reaffirmed **do-not-build**, each with current evidence: L7 method/path + TLS-MITM (srt ships no DSL and calls SNI-only acceptable; OpenShell needs it for creds+inference sandy doesn't do); deny-first default (OpenShell's own friction-mitigation machinery is the proof it's too heavy for a solo tool); the gateway/gRPC/control-plane/K8s platform (OpenShell = ~185k LOC anti-sandy); a driver/backend abstraction (ride Docker `--runtime`, don't build `ComputeDriver`); persistent-service/daemon mode (alice's niche; breaks the ephemeral identity); per-command escape hatches (`dangerouslyDisableSandbox`); cloud teleport / multi-cloud (agentbox's platform axis); Windows (Docker Sandboxes covers it). **Revised:** the "ignore OpenShell" verdict tightens to an active *watch* on the competitive axis (NVIDIA weight), while standing on architecture.

### 6.1 Top implementation sketches (maintainer-could-start-tomorrow)

1. **`SANDY_EGRESS_PROXY` downgrade gate.** *Mechanism:* keep the key passive for the *tightening* direction; if a **workspace** source sets it to `0` (or any value weaker than the effective host/default), route it through the existing passive-privileged approval path (the same sha256-over-KEY=VALUE file). *Tier:* passive-to-tighten, approval-to-loosen (a new sub-case, not a retier — frame as invariant-bugfix). *Failure modes:* headless → fail-closed (drop the downgrade, keep isolation on, warn). *Tests:* `run-tests.sh` new § — workspace `=0` without approval must not disable the proxy; env `export=0` still wins (documented escape). *Spec:* CLAUDE.md Config-tiers + SPEC App. C; fix the `sandy:51` comment.
2. **`.env` protection.** *Mechanism:* at launch, scan `$WORK_DIR` ≤3 levels (skip `node_modules/`, `.venv*/`, `.git/`) for `.env`, `.env.*`, `.env.local`; add matches to the protected-path `:ro` mount list (reuse existing machinery), or mask via a zero-perm bind for read-protection. *Tier:* on by default; `SANDY_ALLOW_ENV_READ=1` passive escape for projects that need it. *Failure modes:* a huge tree — cap scan count + time, log what was skipped. *Tests:* seed `.env`, assert `:ro`/masked in-container; assert `.env.example` untouched. *Spec:* Protected Files + THREAT_MODEL R2b.
3. **Transparent-launch primitive (`SANDY_LAUNCH=raw` / `sandy --exec`).** *Mechanism:* a launch path that `exec`s the agent attached to the current PTY with **no inner tmux, no auto-resume wrapper, verbatim agent args** — generalizing the claude-only `SANDY_REMOTE_CONTROL` no-tmux path (`sandy:2596`) to all agents. Interactive-TUI managers point their launch command at `sandy` and parse the agent's native terminal output (today they hit the tmux-in-tmux wrap at `sandy:2613`). Advertise it in `--print-schema` so managers discover sandy as a drop-in agent-command target. *Tier:* passive (`SANDY_LAUNCH`) + a `--exec` flag. *Failure modes:* multi-agent + raw is incoherent → error (raw is single-agent); auto-resume disabled in raw so managers own session lifecycle. *Tests:* `run-tests.sh §` — raw mode emits no `tmux new-session`; args pass verbatim; `--print-schema` lists it. *Spec:* new CLAUDE.md "Manager integration" section + SPEC App. E launch assembly + the introspection schema. *Strategic note:* this is the single change that converts the 25-tool worktree-manager ecosystem (Superset/Orca/Claude Squad/vibe-kanban/cmux/crystal) from competitors/bystanders into point-and-harden front-ends for sandy.
4. **CC-nested-in-sandy doc + optional seed.** *Mechanism:* document (and optionally seed into the claude pane's `settings.local.json`) `sandbox.enabled:true` + `enableWeakerNestedSandbox:true` + a `sandbox.credentials` deny list covering the ephemeral cred mount + discovered `.env`. *Tier:* doc-only first; opt-in seed `SANDY_CLAUDE_BASH_SANDBOX=1` later. *Failure modes:* documented tool breakage (docker/watchman/Go-TLS) — list them; keep off by default. *Tests:* n/a (doc); later, integration assert the setting lands. *Spec:* new CLAUDE.md subsection + SPEC App. C.
5. **Git credential broker (Family 2).** *Mechanism:* `~/.local/bin/git` shim in-container intercepts push/pull/fetch/clone (flag-whitelist + positional-refusal, ~120 lines bash), pipes over a `$SANDBOX_DIR/broker.sock` bind-mounted Unix socket to a host helper (sandy's existing per-launch-helper + cleanup-trap pattern) that runs `git -C $WORK_DIR <op>`; per-action approval reuses the sha256-approval file, content-bound to argv+repo+branch+commit+diff hash + TTL; optional FIDO2 `verify-required` touch on push. Loft-sh/devpod + the VSCode helper-client↔helper-server socket design are the OSS references to read first. *Tier:* opt-in `SANDY_GIT_BROKER=1`, privileged. *Failure modes:* headless → fail-closed; unknown flag → die; socket absent → error not bypass. *Tests:* pure `run-tests.sh §` for shim whitelist/positional-refusal; integration § for socket round-trip. *Spec:* new "Credential Broker" section + App. E mounts/helpers.
6. **gstack SHA pin.** *Mechanism:* resolve the pack version once, record the SHA in `~/.sandy/.skill_version_gstack`, and **do not** auto-advance to a newer upstream HEAD without an explicit `sandy --update-skill-packs` (or a `SANDY_SKILL_PACK_REF` pin). *Tier:* passive pin key; update is an explicit command. *Failure modes:* first run with no network → hardcoded fallback SHA (already exists). *Tests:* assert no rebuild when upstream HEAD moves but the pin is unchanged. *Spec:* CLAUDE.md Version-Resolution.
